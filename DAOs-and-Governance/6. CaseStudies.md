# DAO Case Studies

## Introduction

This document examines successful DAOs across different categories, analyzing their governance structures, operational models, challenges faced, and lessons learned. These case studies provide practical insights into what works (and what doesn't) in decentralized organizations.

## Protocol DAOs

### MakerDAO

**Overview:**
- Founded: 2014
- Purpose: Govern the DAI stablecoin protocol
- Treasury: $5+ billion
- Token: MKR

**Governance Structure:**

```
MakerDAO Governance:
├── MKR Token Holders (ultimate authority)
├── Delegates (active voters)
├── Core Units (operational teams)
│   ├── Protocol Engineering
│   ├── Risk
│   ├── Oracles
│   ├── Growth
│   └── Real World Finance
└── Emergency Shutdown Module (safety mechanism)
```

**Key Innovations:**

1. **Core Unit Framework**
```python
class CoreUnit:
    def __init__(self, name, mandate, budget):
        self.name = name
        self.mandate = mandate  # Clear scope of work
        self.budget = budget    # Quarterly budget
        self.facilitator = None # Team lead
        self.contributors = []
    
    def quarterly_cycle(self):
        """Core Unit operational cycle"""
        return {
            'month_1': 'Execute work, weekly updates',
            'month_2': 'Continue execution, mid-quarter review',
            'month_3': 'Complete deliverables, prepare next budget',
            'governance': 'Budget proposal for next quarter'
        }
```

2. **Delegation System**
- Recognized delegates with public platforms
- Delegate compensation
- Accountability through regular communication

3. **Emergency Shutdown**
- Last resort safety mechanism
- Requires MKR vote
- Allows DAI holders to redeem collateral

**Challenges:**

1. **Governance Overhead**
- Too many proposals
- Voter fatigue
- Complex technical decisions

**Solution:** Core Units with delegated authority

2. **Centralization Concerns**
- Large MKR holders have significant influence
- Core Unit facilitators have operational control

**Mitigation:** Transparency, multiple Core Units, community oversight

**Lessons Learned:**
- Delegate operational authority to specialized teams
- Balance decentralization with efficiency
- Clear mandates and budgets reduce governance overhead
- Emergency mechanisms are essential for financial protocols

### Uniswap DAO

**Overview:**
- Founded: 2020 (governance launched)
- Purpose: Govern Uniswap DEX protocol
- Treasury: $2.5+ billion
- Token: UNI

**Governance Structure:**

```
Uniswap Governance:
├── UNI Token Holders
├── Delegates (2.5M UNI to propose)
├── Governance Forum (discussion)
├── Snapshot (temperature checks)
├── On-Chain Voting (binding)
└── Timelock (2-day delay)
```

**Key Features:**

1. **High Proposal Threshold**
```solidity
// Uniswap Governor
uint256 public constant PROPOSAL_THRESHOLD = 2_500_000e18; // 2.5M UNI
uint256 public constant QUORUM_VOTES = 40_000_000e18;      // 40M UNI
```

**Rationale:** Prevents spam, ensures serious proposals

2. **Grants Program**
- Uniswap Foundation manages grants
- Community-driven allocation
- Focus on ecosystem growth

3. **Fee Switch Debate**
- Protocol currently charges 0% fee
- Governance can enable fee (0.05%)
- Ongoing discussion about value capture

**Challenges:**

1. **Low Voter Participation**
- Average turnout: 5-10%
- High quorum difficult to reach

**Solution:** Delegation encouraged, lower quorum discussed

2. **Governance Minimalism**
- Few proposals pass
- Conservative approach

**Philosophy:** "Do no harm" - protocol works well, minimal changes needed

**Lessons Learned:**
- High thresholds prevent spam but may limit participation
- Successful protocols may need minimal governance
- Delegation is critical for participation
- Value capture vs growth is ongoing debate

### Compound

**Overview:**
- Founded: 2020 (governance launched)
- Purpose: Govern Compound lending protocol
- Treasury: $500M+
- Token: COMP

**Governance Structure:**

```
Compound Governance:
├── COMP Token Holders
├── Autonomous Proposals (65k COMP threshold)
├── Governance Forum
├── On-Chain Voting
└── Timelock (2-day delay)
```

**Key Innovations:**

1. **First DeFi Governance Token**
- Pioneered "liquidity mining"
- Distributed COMP to users
- Created governance token standard

2. **Autonomous Proposals**
```solidity
// Compound Governor Bravo

    address[] memory targets,
    uint[] memory values,
    string[] memory signatures,
    bytes[] memory calldatas,
    string memory description

        comp.getPriorVotes(msg.sender, sub256(block.number, 1)) > proposalThreshold(),

```

3. **Grants Program**
- Compound Grants Program (CGP)
- Community-driven
- Focus on protocol development and adoption

**Challenges:**

1. **Governance Attack (Proposal 062)**
- Controversial proposal to distribute COMP
- Passed despite community opposition
- Highlighted governance vulnerabilities

**Response:** Increased scrutiny, better communication

2. **Whale Dominance**
- Large holders have significant influence
- Centralization concerns

**Mitigation:** Delegation, community engagement

**Lessons Learned:**
- First-mover advantage in governance tokens
- Liquidity mining can bootstrap participation
- Governance attacks are real threats
- Community vigilance is essential

## Investment DAOs

### The LAO

**Overview:**
- Founded: 2020
- Purpose: For-profit investment DAO
- Structure: Delaware LLC
- Members: Accredited investors only

**Legal Structure:**

```
The LAO Structure:
├── Delaware LLC (legal wrapper)
├── Operating Agreement (governance rules)
├── Members (accredited investors)
├── Service Providers (legal, admin)
└── Smart Contracts (Moloch v2)
```

**Key Features:**

1. **Legal Compliance**
- SEC-compliant structure
- Accredited investor verification
- Securities law compliance
- Tax reporting (K-1s)

2. **Moloch Framework**
```solidity
// Moloch v2 - Ragequit mechanism
function ragequit(uint256 sharesToBurn, uint256 lootToBurn) public {
    _ragequit(msg.sender, sharesToBurn, lootToBurn);
}

function _ragequit(address memberAddress, uint256 sharesToBurn, uint256 lootToBurn) internal {
    // Member can exit with pro-rata share of treasury
    // Protects minority members
}
```

3. **Investment Process**
```markdown
1. Deal sourcing (members or external)
2. Due diligence (members volunteer)
3. Proposal submission (on-chain)
4. Discussion period (7 days)
5. Voting period (7 days)
6. Grace period (7 days for ragequit)
7. Execution (if passed)
```

**Challenges:**

1. **Regulatory Compliance**
- Complex legal requirements
- Ongoing compliance costs
- Limited to accredited investors

2. **Coordination Overhead**
- Due diligence requires effort
- Slow decision-making
- Member engagement varies

**Lessons Learned:**
- Legal structure is essential for investment DAOs
- Ragequit protects minority members
- Accredited investor requirement limits participation
- Professional service providers are necessary

## Grant DAOs

### Gitcoin DAO

**Overview:**
- Founded: 2021 (DAO launched)
- Purpose: Fund public goods
- Mechanism: Quadratic funding
- Token: GTC

**Governance Structure:**

```
Gitcoin DAO:
├── GTC Token Holders
├── Stewards (elected representatives)
├── Workstreams (operational teams)
│   ├── Public Goods
│   ├── Fraud Detection & Defense
│   ├── Moonshot Collective
│   └── DAO Ops
└── Grants Program
```

**Quadratic Funding:**

```python
def quadratic_funding(contributions):
    """
    Calculate quadratic funding match
    
    Formula: Match = (√contribution₁ + √contribution₂ + ... + √contributionₙ)²
    
    This favors projects with many small contributors over
    projects with few large contributors
    """
    sqrt_sum = sum(math.sqrt(c) for c in contributions)
    match_amount = sqrt_sum ** 2
    return match_amount

# Example:
# Project A: 10 contributors × $10 = $100 raised
# Match: (√10 × 10)² = (10 × 3.16)² = $1,000 match
#
# Project B: 1 contributor × $100 = $100 raised  
# Match: (√100)² = 10² = $100 match
#
# Project A gets 10x more matching despite same amount raised!
```

**Key Innovations:**

1. **Quadratic Funding**
- Democratizes funding allocation
- Rewards broad community support
- Reduces whale influence

2. **Sybil Resistance**
- Gitcoin Passport (identity verification)
- Machine learning fraud detection
- Community flagging

3. **Workstream Model**
- Autonomous teams with budgets
- Clear mandates
- Quarterly funding

**Challenges:**

1. **Sybil Attacks**
- Fake accounts to game matching
- Constant cat-and-mouse game

**Solution:** Multi-layered defense (Passport, ML, community)

2. **Sustainability**
- Matching pool funding
- Protocol revenue needed

**Approach:** Gitcoin Grants Protocol (decentralized)

**Lessons Learned:**
- Quadratic funding is powerful for public goods
- Sybil resistance is critical
- Workstream model enables scale
- Sustainability requires protocol revenue

### Moloch DAO

**Overview:**
- Founded: 2019
- Purpose: Fund Ethereum development
- Mechanism: Grants
- Framework: Moloch v1/v2

**Governance Structure:**

```
Moloch DAO:
├── Members (shares)
├── Proposals (funding requests)
├── Voting (simple majority)
├── Grace Period (ragequit window)
└── Guild Kick (remove malicious members)
```

**Key Features:**

1. **Minimalist Design**
```solidity
// Moloch v1 - Extremely simple
contract Moloch {
    mapping(address => uint256) public shares;
    
    function submitProposal(
        address applicant,
        uint256 sharesRequested,
        uint256 tributeOffered
    ) public {}
    
    function submitVote(uint256 proposalIndex, uint8 vote) public {}
    
    function processProposal(uint256 proposalIndex) public {}
    
    function ragequit(uint256 sharesToBurn) public {}
}
```

2. **Ragequit Mechanism**
- Members can exit with pro-rata treasury share
- Protects against malicious proposals
- Enables minority protection

3. **Guild Kick**
- Remove malicious or inactive members
- Requires supermajority
- Returns pro-rata share

**Impact:**
- Funded critical Ethereum infrastructure
- Inspired many DAO frameworks
- Proved minimalist governance works

**Lessons Learned:**
- Simple is often better
- Ragequit is powerful protection
- Focus on mission attracts aligned members
- Small, focused DAOs can be highly effective

## Service DAOs

### RaidGuild

**Overview:**
- Founded: 2020
- Purpose: Web3 development services
- Structure: Unincorporated
- Members: 100+ developers/designers

**Operating Model:**

```
RaidGuild Structure:
├── Members (contributors)
├── Raids (client projects)
├── Cleric (project manager)
├── Champions (client relations)
└── Treasury (shared revenue)
```

**Revenue Model:**

```python
class RaidGuildEconomics:
    def __init__(self):
        self.revenue_split = {
            'raid_party': 0.70,      # 70% to project team
            'treasury': 0.20,         # 20% to DAO treasury
            'client_acquisition': 0.10 # 10% to champion
        }
    
    def calculate_payment(self, project_revenue):
        """Calculate payment distribution"""
        return {
            'raid_party': project_revenue * self.revenue_split['raid_party'],
            'treasury': project_revenue * self.revenue_split['treasury'],
            'champion': project_revenue * self.revenue_split['client_acquisition']
        }
```

**Membership Levels:**

```markdown
1. **Apprentice**
   - New member
   - Learning phase
   - Can join raids

2. **Member**
   - Completed raids
   - Full voting rights
   - Can lead raids

3. **Champion**
   - Client relations
   - Business development
   - Earn acquisition fees
```

**Lessons Learned:**
- Service DAOs can generate revenue
- Clear project structure is essential
- Revenue sharing aligns incentives
- Reputation matters for client work

## Common Success Factors

### 1. Clear Purpose

All successful DAOs have a well-defined mission:
- MakerDAO: Stable decentralized currency
- Uniswap: Decentralized exchange
- Gitcoin: Fund public goods
- FWB: Creator community

### 2. Appropriate Structure

Structure matches purpose:
- Protocol DAOs: Delegated authority (Core Units, Workstreams)
- Investment DAOs: Legal wrappers + Moloch
- Grant DAOs: Quadratic funding + Workstreams
- Social DAOs: Curated membership
- Service DAOs: Project-based teams

### 3. Aligned Incentives

Token economics align with goals:
- Governance tokens for protocol DAOs
- Shares for investment DAOs
- Membership tokens for social DAOs
- Revenue sharing for service DAOs

### 4. Progressive Decentralization

Most started centralized, gradually decentralized:
```
Phase 1: Core team builds product
Phase 2: Community forms
Phase 3: Governance token launches
Phase 4: Progressive authority transfer
Phase 5: Full decentralization
```

### 5. Professional Operations

Successful DAOs have professional operations:
- Clear processes
- Documentation
- Regular communication
- Accountability mechanisms
- Professional tools

## Common Failure Patterns

### 1. Governance Theater

Appearance of decentralization without substance:
- Token holders can't effect change
- Core team retains control
- Proposals are rubber stamps

### 2. Plutocracy

Wealth concentration leads to power concentration:
- Whales control decisions
- Community disengagement
- Centralization despite decentralized structure

### 3. Coordination Failure

Unable to make decisions or execute:
- Too much decentralization
- No clear authority
- Endless discussion, no action

### 4. Legal/Regulatory Issues

Ignoring legal reality:
- Securities violations
- Tax non-compliance
- Liability exposure

### 5. Treasury Mismanagement

Poor financial management:
- Insufficient runway
- Risky investments
- No diversification
- Lack of transparency

## Key Takeaways

1. **One Size Doesn't Fit All**
   - Different DAO types need different structures
   - Match governance to purpose

2. **Balance Decentralization and Efficiency**
   - Pure decentralization is often inefficient
   - Delegate operational authority
   - Reserve governance for key decisions

3. **Legal Structure Matters**
   - Especially for investment and service DAOs
   - Protects members
   - Enables real-world operations

4. **Progressive Decentralization Works**
   - Start centralized, gradually decentralize
   - Build product first, governance second
   - Transfer authority over time

5. **Professional Operations Are Essential**
   - DAOs need structure and processes
   - Documentation and communication matter
   - Tools and systems enable scale

6. **Community Is Everything**
   - Aligned members are the foundation
   - Culture matters
   - Engagement requires effort

7. **Sustainability Requires Revenue**
   - Treasury depletion is real risk
   - Protocol revenue or services needed
   - Grants are not sustainable long-term

## Conclusion

These case studies demonstrate that DAOs can work across various use cases, from protocol governance to investment to social coordination. Success requires:

- Clear purpose and mission
- Appropriate governance structure
- Aligned incentives
- Professional operations
- Legal compliance
- Community engagement
- Financial sustainability

The DAO space continues to evolve rapidly, with new models and innovations emerging constantly. These case studies provide a foundation for understanding what works, but experimentation and adaptation remain essential.
