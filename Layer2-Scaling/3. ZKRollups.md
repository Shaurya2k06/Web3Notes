# ZK Rollups (Zero-Knowledge Rollups)

## Overview

ZK Rollups are Layer 2 scaling solutions that use zero-knowledge proofs to verify the correctness of off-chain computations. Unlike Optimistic Rollups that assume validity, ZK Rollups provide cryptographic proof of validity for every state transition.

### Core Principle

```
Validity Proof Approach:
"Prove transactions are valid cryptographically"

Process:
1. Execute transactions off-chain
2. Generate zero-knowledge proof of correct execution
3. Post proof + compressed data to L1
4. L1 verifies proof (cheap)
5. Immediate finality (no challenge period)
```

## Architecture

### Components

**1. Sequencer/Operator**
- Executes transactions off-chain
- Generates batches
- Produces validity proofs
- Posts to L1

**2. Prover**
- Generates zero-knowledge proofs
- Computationally intensive
- Can be decentralized
- Specialized hardware often used

**3. Verifier Contract (L1)**
- Verifies validity proofs
- Updates state root
- Handles deposits/withdrawals
- Constant gas cost regardless of batch size

**4. State Tree**
- Merkle tree of account states
- Updated with each batch
- Root posted to L1

### System Flow

```python
class ZKRollup:
    def __init__(self):
        self.state_root = bytes(32)
        self.pending_transactions = []
        
    def execute_batch(self, transactions):
        """
        Execute batch and generate proof
        """
        # Execute transactions off-chain
        new_state, state_delta = self.execute_transactions(transactions)
        
        # Generate validity proof
        proof = self.generate_proof(
            old_state=self.state_root,
            transactions=transactions,
            new_state=new_state
        )
        
        # Compress transaction data
        compressed_data = self.compress_transactions(transactions)
        
        return {
            'new_state_root': new_state,
            'proof': proof,
            'compressed_data': compressed_data
        }
    
    def submit_batch_to_l1(self, batch_data):
        """
        Submit batch with proof to L1
        """
        # L1 contract verifies proof
        # If valid, updates state root immediately
        # No challenge period needed
        
        tx = self.l1_contract.submitBatch(
            newStateRoot=batch_data['new_state_root'],
            proof=batch_data['proof'],
            compressedData=batch_data['compressed_data']
        )
        
        return tx
    
    def generate_proof(self, old_state, transactions, new_state):
        """
        Generate zero-knowledge proof
        This is computationally expensive
        """
        # Create circuit that proves:
        # "I know transactions T such that
        #  executing T on old_state produces new_state"
        
        circuit = self.create_circuit(old_state, transactions, new_state)
        proof = self.prove(circuit)
        
        return proof
```

## Zero-Knowledge Proof Systems

### zk-SNARKs

**Characteristics:**
- Small proof size (~200 bytes)
- Fast verification (~5ms)
- Requires trusted setup
- Quantum vulnerable

**Used by:**
- zkSync Era (Boojum)
- Scroll
- Polygon zkEVM (Type 3)

class zkSNARK:
    def __init__(self):
        self.proving_key, self.verification_key = self.trusted_setup()
    
    def trusted_setup(self):
        """
        Generate proving and verification keys
        Requires trusted ceremony

        # Generate toxic waste
        tau = random_field_element()

        pk = generate_proving_key(tau)
        vk = generate_verification_key(tau)

        del tau

    def prove(self, circuit, witness):
        """
        Generate proof for circuit with witness

        Verify proof (fast, constant time)

        return groth16_verify(proof, public_inputs, self.verification_key)

### zk-STARKs

**Characteristics:**
- Larger proof size (~100-200 KB)
- Slower verification (~100ms)
- No trusted setup
- Quantum resistant
- Transparent

**Used by:**
- StarkNet
- StarkEx (dYdX, Immutable X)

```python
# Simplified zk-STARK proof generation
class zkSTARK:
    def __init__(self):
        # No trusted setup needed
        pass
    
    def prove(self, circuit, witness):
        """
        Generate STARK proof
        """
        # Convert to AIR (Algebraic Intermediate Representation)
        air = circuit_to_air(circuit)
        
        # Generate execution trace
        trace = generate_trace(air, witness)
        
        # Apply FRI (Fast Reed-Solomon Interactive Oracle Proof)
        proof = fri_prove(trace)
        
        return proof
    
    def verify(self, proof, public_inputs):
        """
        Verify STARK proof
        """
        return fri_verify(proof, public_inputs)

```

## EVM Compatibility Levels

ZK Rollups face challenges in proving EVM execution due to its complexity.

### Type 1: Ethereum-Equivalent

**Definition**: Fully equivalent to Ethereum, no modifications.

**Characteristics:**
- Can verify Ethereum blocks
- Slowest proof generation
- Perfect compatibility
- No code changes needed

**Example**: Taiko (in development)

**Proof Time**: Hours per block

### Type 2: EVM-Equivalent

**Definition**: Equivalent to EVM, minor modifications to data structures.

**Characteristics:**
- Modify gas costs slightly
- Change some data structures for efficiency
- Nearly perfect compatibility
- Minimal code changes

**Example**: Polygon zkEVM, Scroll

**Proof Time**: ~30 minutes per block

### Type 3: EVM-Compatible

**Definition**: Compatible with most EVM applications, some modifications.

**Characteristics:**
- Remove some precompiles
- Modify some opcodes
- Most dApps work without changes
- Some edge cases may break

**Example**: zkSync Era, Linea

**Proof Time**: ~10 minutes per block

### Type 4: High-Level Language Compatible

**Definition**: Compile from Solidity/Vyper to custom VM.

**Characteristics:**
- Custom VM optimized for ZK proofs
- Fast proof generation
- May require code modifications
- Different bytecode

**Example**: zkSync Era (partially), StarkNet (Cairo)

**Proof Time**: Minutes per block

## Proof Generation

### Circuit Design

```python
class ZKCircuit:
    def __init__(self):
        self.constraints = []
        
    def add_constraint(self, a, b, c):
        """
        Add constraint: a * b = c
        """
        self.constraints.append((a, b, c))
    
    def verify_transaction(self, tx, old_state, new_state):
        """
        Create circuit to verify transaction execution
        """
        # Constraint 1: Signature verification
        self.verify_signature(tx.signature, tx.hash, tx.from_address)
        
        # Constraint 2: Balance check
        self.add_constraint(
            old_state.balance[tx.from_address] >= tx.value,
            1,
            1
        )
        
        # Constraint 3: Nonce check
        self.add_constraint(
            old_state.nonce[tx.from_address],
            1,
            tx.nonce
        )
        
        # Constraint 4: State transition
        self.verify_state_transition(tx, old_state, new_state)
        
        return self.constraints
    
    def verify_state_transition(self, tx, old_state, new_state):
        """
        Verify state changes are correct
        """
        # From balance decreases
        self.add_constraint(
            old_state.balance[tx.from_address] - tx.value,
            1,
            new_state.balance[tx.from_address]
        )
        
        # To balance increases
        self.add_constraint(
            old_state.balance[tx.to_address] + tx.value,
            1,
            new_state.balance[tx.to_address]
        )
        
        # Nonce increases
        self.add_constraint(
            old_state.nonce[tx.from_address] + 1,
            1,
            new_state.nonce[tx.from_address]
        )
```

### Recursive Proofs

Prove multiple batches with a single proof.

```python
class RecursiveProver:
    def __init__(self):
        self.batch_proofs = []
    
    def prove_batch(self, batch):
        """Generate proof for single batch"""
        proof = generate_proof(batch)
        self.batch_proofs.append(proof)
        return proof
    
    def aggregate_proofs(self, proofs):
        """
        Generate single proof that verifies multiple proofs
        """
        # Create circuit that verifies all input proofs
        aggregation_circuit = self.create_aggregation_circuit(proofs)
        
        # Generate proof of proof verification
        aggregated_proof = generate_proof(aggregation_circuit)
        
        return aggregated_proof
    
    def create_aggregation_circuit(self, proofs):
        """
        Circuit that proves: "I verified all these proofs"
        """
        circuit = Circuit()
        
        for proof in proofs:
            # Add verification constraints for each proof
            circuit.add_proof_verification(proof)
        
        return circuit

# Example: Aggregate 100 batch proofs into 1
prover = RecursiveProver()
batch_proofs = [prover.prove_batch(batch) for batch in batches]
final_proof = prover.aggregate_proofs(batch_proofs)

print(f"Aggregated {len(batch_proofs)} proofs into 1")
print(f"L1 verification cost: constant (regardless of batch count)")
```

## Performance Optimization

### Batch Size Optimization

```python
def optimize_batch_size(
    proof_generation_cost,
    proof_verification_cost,
    tx_data_cost_per_byte,
    avg_tx_size_bytes
):
    """
    Find optimal batch size to minimize cost per transaction
    """
    batch_sizes = range(100, 10000, 100)
    results = []
    
    for batch_size in batch_sizes:
        # Fixed costs (amortized over batch)
        proof_cost_per_tx = (proof_generation_cost + proof_verification_cost) / batch_size
        
        # Variable costs (per transaction)
        data_cost_per_tx = tx_data_cost_per_byte * avg_tx_size_bytes
        
        # Total cost per transaction
        total_cost_per_tx = proof_cost_per_tx + data_cost_per_tx
        
        results.append({
            'batch_size': batch_size,
            'cost_per_tx': total_cost_per_tx
        })
    
    # Find minimum
    optimal = min(results, key=lambda x: x['cost_per_tx'])
    return optimal

# Example
optimal = optimize_batch_size(
    proof_generation_cost=1000,  # $1000 to generate proof
    proof_verification_cost=50,   # $50 to verify on L1
    tx_data_cost_per_byte=0.01,  # $0.01 per byte
    avg_tx_size_bytes=100         # 100 bytes per tx
)

print(f"Optimal batch size: {optimal['batch_size']} transactions")
print(f"Cost per transaction: ${optimal['cost_per_tx']:.4f}")
```

## Conclusion

ZK Rollups represent the cutting edge of Layer 2 scaling, offering instant finality and strong security guarantees through cryptographic proofs. While more complex to implement than Optimistic Rollups, they provide superior user experience with no withdrawal delays.

**Key Takeaways:**
- Use validity proofs instead of fraud proofs
- Instant finality (no challenge period)
- More complex implementation
- Two main proof systems: SNARKs and STARKs
- EVM compatibility is challenging but improving
- Major implementations: zkSync, StarkNet, Polygon zkEVM, Scroll
- Suitable for high-value transfers and applications requiring fast finality
