# Optimistic Rollups

## Overview

Optimistic Rollups are Layer 2 scaling solutions that assume transactions are valid by default ("optimistic") and only run computation on-chain if a transaction is challenged through a fraud proof. This approach significantly increases throughput while maintaining Ethereum's security guarantees.

### Core Principle

```
Optimistic Assumption:
"Transactions are valid unless proven otherwise"

Process:
1. Execute transactions off-chain
2. Post transaction data to L1
3. Wait for challenge period (7 days)
4. If no valid challenge, finalize state
5. If challenged, run fraud proof on L1
```

## Architecture

### Components

**1. Sequencer**
- Orders and executes transactions
- Batches transactions for L1 submission
- Provides instant soft confirmations
- Currently centralized in most implementations

**2. State Root**
- Merkle root of L2 state
- Posted to L1 periodically
- Represents claimed state after batch execution

**3. Transaction Data**
- Compressed transaction data posted to L1
- Enables anyone to reconstruct L2 state
- Critical for data availability

**4. Fraud Proof System**
- Mechanism to challenge invalid state transitions
- Runs disputed computation on L1
- Slashes malicious proposers

### System Flow

```python
class OptimisticRollup:
    def __init__(self):
        self.state = {}
        self.pending_batches = []
        self.challenge_period = 7 * 24 * 60 * 60  # 7 days in seconds
        
    def submit_batch(self, transactions, new_state_root):
        """
        Sequencer submits batch to L1
        """
        batch = {
            'transactions': transactions,
            'old_state_root': self.get_current_state_root(),
            'new_state_root': new_state_root,
            'timestamp': time.time(),
            'status': 'pending'
        }
        
        # Post to L1
        self.post_to_l1(batch)
        self.pending_batches.append(batch)
        
        return batch
    
    def challenge_batch(self, batch_id, fraud_proof):
        """
        Verifier challenges invalid batch
        """
        batch = self.pending_batches[batch_id]
        
        # Verify fraud proof on L1
        if self.verify_fraud_proof(batch, fraud_proof):
            # Slash sequencer
            self.slash_sequencer(batch['sequencer'])
            # Revert state
            self.revert_to_state(batch['old_state_root'])
            return True
        
        return False
    
    def finalize_batch(self, batch_id):
        """
        Finalize batch after challenge period
        """
        batch = self.pending_batches[batch_id]
        
        if time.time() - batch['timestamp'] >= self.challenge_period:
            batch['status'] = 'finalized'
            self.state = batch['new_state_root']
            return True
        
        return False
```

## Fraud Proof Mechanisms

### Interactive Fraud Proofs

Most Optimistic Rollups use interactive fraud proofs to minimize on-chain computation.

**Process:**

```
1. Challenger disputes state transition
2. Binary search to find first incorrect step
3. Execute single step on L1
4. Verify result matches claimed state

Example:
Batch has 1000 transactions
- Round 1: Dispute at tx 500
- Round 2: Dispute at tx 250
- Round 3: Dispute at tx 125
- ...
- Round 10: Execute single tx on L1
```

**Implementation:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract InteractiveFraudProof {
    struct Dispute {
        address challenger;
        address sequencer;
        bytes32 stateRoot;
        uint256 startBlock;
        uint256 endBlock;
        uint256 currentStep;
        bool resolved;
    }
    
    mapping(uint256 => Dispute) public disputes;
    uint256 public disputeCount;
    
    event DisputeInitiated(
        uint256 indexed disputeId,
        address indexed challenger,
        bytes32 stateRoot
    );
    
    event DisputeResolved(
        uint256 indexed disputeId,
        bool challengerWon
    );
    
    function initiateDispute(
        bytes32 stateRoot,
        uint256 startBlock,
        uint256 endBlock
    ) external returns (uint256) {
        uint256 disputeId = disputeCount++;
        
        disputes[disputeId] = Dispute({
            challenger: msg.sender,
            sequencer: getSequencer(stateRoot),
            stateRoot: stateRoot,
            startBlock: startBlock,
            endBlock: endBlock,
            currentStep: 0,
            resolved: false
        });
        
        emit DisputeInitiated(disputeId, msg.sender, stateRoot);
        return disputeId;
    }
    
    function bisectExecution(
        uint256 disputeId,
        uint256 midpoint,
        bytes32 midpointStateRoot
    ) external {
        Dispute storage dispute = disputes[disputeId];
        require(!dispute.resolved, "Dispute already resolved");
        require(
            msg.sender == dispute.challenger || msg.sender == dispute.sequencer,
            "Not a party to dispute"
        );
        
        // Binary search logic
        if (verifyMidpointState(midpointStateRoot, midpoint)) {
            // Correct up to midpoint, search second half
            dispute.startBlock = midpoint;
        } else {
            // Incorrect before midpoint, search first half
            dispute.endBlock = midpoint;
        }
        
        dispute.currentStep++;
        
        // If narrowed to single transaction, execute on L1
        if (dispute.endBlock - dispute.startBlock == 1) {
            resolveDispute(disputeId);
        }
    }
    
    function resolveDispute(uint256 disputeId) internal {
        Dispute storage dispute = disputes[disputeId];
        
        // Execute single transaction on L1
        bytes32 computedState = executeTransactionOnL1(
            dispute.startBlock,
            dispute.stateRoot
        );
        
        bool challengerWon = (computedState != dispute.stateRoot);
        
        if (challengerWon) {
            // Slash sequencer
            slashSequencer(dispute.sequencer);
            // Reward challenger
            rewardChallenger(dispute.challenger);
        } else {
            // Slash challenger (frivolous challenge)
            slashChallenger(dispute.challenger);
        }
        
        dispute.resolved = true;
        emit DisputeResolved(disputeId, challengerWon);
    }
    
    function executeTransactionOnL1(
        uint256 blockNumber,
        bytes32 preState
    ) internal pure returns (bytes32) {
        // Execute single transaction and return resulting state
        // This is the only expensive on-chain computation
        // Implementation depends on VM being used
        return keccak256(abi.encodePacked(preState, blockNumber));
    }
    
    function verifyMidpointState(
        bytes32 stateRoot,
        uint256 blockNumber
    ) internal view returns (bool) {

        return true; // Simplified
    }
    
    function getSequencer(bytes32 stateRoot) internal view returns (address) {
        // Get sequencer who proposed this state
        return address(0); // Simplified
    }
    
    function slashSequencer(address sequencer) internal {
        // Slash sequencer's bond
    }
    
    function rewardChallenger(address challenger) internal {
```

### Non-Interactive Fraud Proofs

Alternative approach: Submit complete fraud proof in single transaction.

**Advantages:**
- Faster resolution
- No back-and-forth required
- Simpler implementation

**Disadvantages:**
- Higher gas costs
- Limited by L1 block gas limit
- Less efficient for complex disputes

```python
        """
        Compress transaction batch for L1 posting
        """
        compressed = bytearray()
        
        for tx in transactions:
            # Use compact encoding
            compressed.extend(self.compress_transaction(tx))
        
        return bytes(compressed)
    
    def compress_transaction(self, tx):
        """
        Compress single transaction
        """
        compressed = bytearray()
        
        # Nonce: Use delta encoding
        compressed.extend(self.encode_varint(tx['nonce_delta']))
        
        # Gas: Use compact representation
        compressed.extend(self.encode_compact_gas(tx['gas']))
        
        # To address: Use address table index
        compressed.extend(self.encode_address_index(tx['to']))
        
        # Value: Use compact representation
        compressed.extend(self.encode_compact_value(tx['value']))
        
        # Data: Compress calldata
        compressed.extend(self.compress_calldata(tx['data']))
        
        # Signature: Use compact signature (v, r, s)
        compressed.extend(self.compress_signature(tx['signature']))
        
        return bytes(compressed)
    
    def encode_varint(self, value):
        """Variable-length integer encoding"""
        result = bytearray()
        while value > 127:
            result.append((value & 0x7F) | 0x80)
            value >>= 7
        result.append(value & 0x7F)
        return bytes(result)

        """Compact gas encoding"""
```

## Sequencer Design

### Centralized Sequencer (Current)

Most Optimistic Rollups currently use a centralized sequencer for simplicity and performance.

**Advantages:**
- Fast transaction confirmation
- Efficient batching
- Simple implementation
- No MEV competition

**Disadvantages:**
- Single point of failure
- Censorship risk
- Trust assumption
- Centralization concern

**Mitigation:**
```solidity
contract CentralizedSequencerWithFallback {
    address public sequencer;
    uint256 public lastBatchTime;
    uint256 public constant MAX_BATCH_DELAY = 1 hours;
    
    function submitBatch(bytes calldata batch) external {
        require(msg.sender == sequencer, "Not sequencer");
        processBatch(batch);
        lastBatchTime = block.timestamp;
    }
    
    function forceIncludeTx(bytes calldata tx) external {
        // If sequencer hasn't submitted batch in MAX_BATCH_DELAY,
        // anyone can force include their transaction
        require(
            block.timestamp - lastBatchTime > MAX_BATCH_DELAY,
            "Sequencer is active"
        );
        
        processTransaction(tx);
    }
}
```

### Decentralized Sequencer (Future)

**Approaches:**

1. **Rotating Sequencer**
```python
class RotatingSequencer:
    def __init__(self, validators):
        self.validators = validators
        self.current_epoch = 0
    
    def get_current_sequencer(self):
        """Rotate sequencer each epoch"""
        index = self.current_epoch % len(self.validators)
        return self.validators[index]
    
    def advance_epoch(self):
        self.current_epoch += 1
```

2. **Auction-Based**
```solidity
contract SequencerAuction {
    struct Bid {
        address bidder;
        uint256 amount;
    }
    
    mapping(uint256 => Bid) public winningBids;
    
    function bidForSequencerRole(uint256 epoch) external payable {
        require(msg.value > winningBids[epoch].amount, "Bid too low");
        
        // Refund previous bidder
        if (winningBids[epoch].bidder != address(0)) {
            payable(winningBids[epoch].bidder).transfer(winningBids[epoch].amount);
        }
        
        winningBids[epoch] = Bid({
            bidder: msg.sender,
            amount: msg.value
        });
    }
}
```

3. **Shared Sequencing**
- Multiple rollups share sequencer set
- Atomic cross-rollup transactions
- Examples: Espresso, Astria

## Major Optimistic Rollup Implementations

### Arbitrum

**Key Features:**
- Multi-round interactive fraud proofs
- Custom AVM (Arbitrum Virtual Machine)
- Nitro upgrade (WASM-based execution)
- Anytrust chains (data availability committees)

**Architecture:**
```
Arbitrum One (Mainnet):
├── Sequencer (Offchain Labs operated)
├── Validators (Permissionless)
├── Fraud Proof System (Interactive)
└── Data Availability (L1 Ethereum)

Arbitrum Nova (AnyTrust):
├── Sequencer (Offchain Labs operated)
├── Data Availability Committee (7 members, 6-of-7 threshold)
├── Fallback to L1 if DAC fails
└── Lower costs, slight trust assumption
```

**Performance:**
- TPS: ~4,000
- Block time: ~0.25 seconds
- Finality: 7 days (L1 withdrawal)
- Cost: ~90% cheaper than L1

### Optimism

**Key Features:**
- Single-round fraud proofs (planned)
- EVM equivalence (minimal modifications)
- OP Stack (modular rollup framework)
- Superchain vision (network of OP Stack chains)

**Architecture:**
```
Optimism Mainnet:
├── Sequencer (Optimism Foundation operated)
├── Proposers (Submit state roots)
├── Challengers (Submit fraud proofs)
└── Data Availability (L1 Ethereum + EIP-4844 blobs)
```

**Performance:**
- TPS: ~2,000
- Block time: ~2 seconds
- Finality: 7 days (L1 withdrawal)
- Cost: ~90% cheaper than L1

## Security Considerations

### Attack Vectors

**1. Sequencer Censorship**
```
Mitigation:
- Force inclusion mechanism
- Decentralized sequencer (future)
- Multiple sequencer fallback
```

**2. Data Withholding**
```
Mitigation:
- All data posted to L1
- Anyone can reconstruct state
- Data availability sampling (future)
```

**3. Invalid State Transition**
```
Mitigation:
- Fraud proof system
- Economic incentives for challengers
- Slashing for malicious proposers
```

**4. Delay Attacks**
```
Mitigation:
- Challenge period timeout
- Multiple verifiers
- Automated fraud proof generation
```

### Economic Security

```python
def calculate_security_budget(tvl, challenge_period_days):
    """
    Calculate required security budget for Optimistic Rollup
    """
    # Attacker profit from stealing TVL
    attacker_profit = tvl
    
    # Cost to bribe all verifiers
    # Assume need to bribe all verifiers for challenge_period
    verifier_count = 100  # Assume 100 active verifiers
    daily_verifier_cost = 1000  # $1000/day per verifier
    
    bribery_cost = verifier_count * daily_verifier_cost * challenge_period_days
    
    # Attack is profitable if profit > cost
    is_secure = bribery_cost > attacker_profit
    
    return {
        'tvl': tvl,
        'bribery_cost': bribery_cost,
        'is_secure': is_secure,
        'security_margin': bribery_cost / attacker_profit if attacker_profit > 0 else float('inf')
    }

# Example
result = calculate_security_budget(tvl=1_000_000_000, challenge_period_days=7)
print(f"TVL: ${result['tvl']:,}")
print(f"Bribery Cost: ${result['bribery_cost']:,}")
print(f"Secure: {result['is_secure']}")
print(f"Security Margin: {result['security_margin']:.2f}x")
```

## Conclusion

Optimistic Rollups provide a practical scaling solution that balances performance, security, and EVM compatibility. While they require a challenge period for withdrawals, they offer significant cost savings and throughput improvements over Layer 1.

**Key Takeaways:**
- Assume validity, verify on challenge (optimistic approach)
- Interactive fraud proofs minimize on-chain computation
- 7-day challenge period for security
- Full EVM compatibility enables easy migration
- Centralized sequencers currently, decentralization planned
- Major implementations: Arbitrum, Optimism, Base
- Suitable for general-purpose dApps and DeFi
